# 介绍
本篇文章主要介绍大模型推理场景下，如何通过 `ait` 工具来进行一键式精度比对。本文采取通俗易懂的语言去尝试解释复杂的定义，对于初学者而言也是十分友好的。

本文先是通过一个简单的数学问题入手，来让读者感受一下精度比对工具的使用场景和基本作用。让读者能够对大模型推理有一个初步的认识和感受。通过这个简单案例来类比实际生产场景，然后详细介绍 `ait llm dump` 的实际功能以及如何去分析它的结果。之后将介绍 `ait llm compare` 和它的使用场景。最后介绍一个案例，帮助读者进一步巩固所学。如果对于 `ait` 工具已有经验，可以直接通过目录进行跳转到感兴趣的章节。

# 简单案例
<a id="简单案例"></a>
此案例针对于对**完全不了解**大模型推理和 `ait` 工具的读者。如果对于这块儿还是比较模糊，还是请有耐心地看完。

类比于实际的大模型业务场景，我们定义两个简单的函数 $f_1(x)$, $f_2(x)$，如下：

$$
    \begin{align}
        f_1(x) = 2x + 1 \\
        f_2(x) = 3x + 2.
    \end{align}
$$
我们将这个两个函数做一个反复嵌套，这样就得到一个另外一个函数，我称之为模型：
$$
    m_1(x) = 4 \cdot f_2(f_1(f_2(f_1(x)))) + 3.
$$
现在考虑到业务场景需求，我们需要将此模型中的两个函数替换成另外两个函数，做一个函数的映射，换言之，$f_1(x) \longrightarrow g_1(x)$，$f_2(x) \longrightarrow g_2(x)$。这两个替换的函数 $g_1(x), g_2(x)$，我们被告知其功能和原函数 $f_1(x), f_2(x)$ 是一样的，且更亲和我们的业务场景。那么此时我们拥有了另一个模型，又名迁移后的模型
$$
    m_2(x) = 4 \cdot g_2(g_1(g_2(g_1(x)))) + 3.
$$
现在有了一个我们的模型，我们想要知道，它的产出是不是**真的**和原模型保持一致。毕竟，不能人家说功能一样就真的一样。

因此，我们打算测试一下。测试方式如下：

1. 随便选一个数字 $x = 2$ 作为输入，让两个模型进行运算，如果结果不一致，则说明存在问题
2. 如果结果一致，我们又额外准备了 `1000` 个不重复数字 $x$ 和 `1000` 个，按照模型的构造，自己手算或者珠心算的答案当作数据集，让这个两个模型进行逐个测试，最后根据答对的题目个数统计分数。如果 $m_2(x)$ 的成绩比 $m_1(x)$ 相差无几，那么我们认为我们的模型还是很不错的。
 
然而，我们在进行第一个测试时就发现问题了。比对结果发现，$m_1(2) = 431$ 但是 $m_2(2) = 171$。通过比较这两个结果，我们可以断定，映射之后的模型 $m_2(x)$ 有精度问题。

出现了问题是好事，这意味着我们替换的函数可能存在一些问题，那么解决了不就好了嘛。然而，我们只知道输出结果不一致，对于中间的计算结果，我们无从得知。如果在计算 $g_2(f_1(2))$ 的时候就已经和 $m_1$ 不一致了，那么这个误差后续肯定会被无限传播和放大，最终得到错误的结果是必然的。

为了找到问题的源头，我们得想办法储存中间的数据。一筹莫展之际，我们突然发现，有一个名叫 `ait dump` 的工具居然可以帮助我们获得中间的结果！此时，我们的测试模型是 $m_2(x)$，我们的标杆模型就是 $m_1(x)$。我们在此工具的帮助下，将两个模型的每个中间结果都保存了下来，如下：

<center>

| 第一层 | $f_1(x)$ | $g_1(x)$ |
| :-: | :----: | :----: |
| param | 2 | 2 |
| retval | 5 | 5 |

| 第二层 | $f_2(x)$ | $g_2(x)$ |
| :-: | :----: | :----: |
| param | 5 | 5 |
| retval | 17 | $\boxed{10}$ |

| 第三层 | $f_1(x)$ | $g_1(x)$ |
| :-: | :----: | :----: |
| param | 17 | $\boxed{10}$ |
| retval | 35 | $\boxed{21}$ |

| 第四层 | $f_1(x)$ | $f_2(x)$ |
| :-: | :----: | :----: |
| param | 35 | $\boxed{21}$ |
| retval | 107 | $\boxed{42}$ |

| 第五层 | $m_1(x)$ | $m_2(x)$ |
| :-: | :----: | :----: |
| param | 53 | $\boxed{42}$ |
| retval | 431 | $\boxed{171}$ |

<caption>Table 1. 使用工具储存的中间计算结果。每一层代表一个函数。param 是函数的参数，retval 是函数的返回值 </caption>

</center>

这个表格将两个模型的每一个中间的计算结果，都进行了保存。我们通过肉眼比较发现，在第二层，计算 $g_2(x)$ 的时候，我们的结果与标杆的 $f_2(x)$ 相比，就已经出现了问题。之后这个问题经过不断地放大，最后导致我们的模型输出存在问题。大功告成！我们定位到了出现问题的映射函数！

# 实际场景
<a id="实际场景"></a>

在上一个案例中，我们定位到了出现问题的函数。有可能是该函数在实现上有些许出入，从而导致对于相同的输入参数，其结果出现了问题。那么在实际的大模型推理场景中，上述的函数又被称为 **算子（Operation，缩写 op）**。函数的入参，在大模型推理场景下，叫做算子的 `in tensor`；函数的返回值，叫做算子的 `out tensor`。所以，大模型的精度比对，就是通过比较每个算子的 `in tensor` 和 `out tensor`，发现存在问题的算子。而不管是中间数据储存，还是比较结果定位到问题算子，都可以通过我们的 `ait` 精度比对工具来进行出色地完成。

在使用精度对比工具之前，首先我们需要确定，模型是否**真的**存在精度问题。回想我们在[简单案例](#简单案例)中的定位过程，我们在使用精度对比工具之前，首先是源于发现了一个 $x = 2$，能够让 $m_1(x) \ne m_2(x)$，也就是让两个模型的输出不一致。在实际的业务场景中，我们称 $m_1(x)$ 为标杆模型，$m_2(x)$ 为测试模型，$x$ 被叫做 **bad case**。

如果我们找不到一个 `bad case`，使得测试模型和标杆模型输出结果不一致，那么就无法证明这个测试模型存在精度问题。此时的测试模型可以不严谨地被认为不存在精度问题。不存在精度问题，使用精度比对工具就毫无意义，模型直接交付就行。注意，我们说测试模型不存在精度问题是相较于标杆模型的表现而言。

## 发现 bad case
那么，如何发现 `bad case` 呢？发现 `bad case` 有哪些步骤呢？ 一般来说，有两种使用最多的方案：

1. 单精度测试（single precision）
2. 数据集测试（dataset）

如果这两种方案都无法发现 `bad case`，一般就认为模型不存在精度问题了。

### 单精度测试

在[简单案例](#简单案例)中，我们发现测试模型存在问题，是源自于一个输入 $x = 2$。我们发现在这个输入下，两个模型的结果出现分歧。由于我们无条件相信标杆模型，所以我们认为测试模型的结果存在问题。这个就叫单精度测试。这个 $x$ 的选择，是毫无逻辑的，是随机的。

在大模型场景中也是如此，不过存在些许差异。我们同样人为的构思一个 $x$，这个输入又被称为提示词（prompt），提问（query）。然后，让大模型根据这个输入进行回答，再比较两个模型的结果。比如：`Where is the capital of China?`，或者 `华为的董事长是谁？`，两个模型根据同一个提示词作出回应，产生结果。

这里需要注意的是，在[简单案例](#简单案例)中，模型的输出结果是存在**标准答案**的，因为只包含纯数字的计算。但是大模型的回答是一句话，如果给模型开放性的题目，是不存在标准答案的。因此，我们单精度测试只是对模型的一个初步筛查，判断模型是否存在胡言乱语的情况。

还存在 `atb` 算子没有接入成功的问题，输出都是 `0` 或者 `NaN`。比如，有时候，模型会出现下列回应：（需要专业人士点评）
```sh
问：Where is the capital of China?

答：<unk><unk><unk><unk><unk><unk><unk><unk>

# Alternatively, sometime it replies
答：월월월월월월월월월월월월월월월월월월월월월월월월월월월월월월월월
```
此时不应该直接使用精度比对工具，而是先去检查模型运行本身存不存在报错，检查日志。

### 数据集测试

如果单精度测试发现不了问题，那么就通过数据集来进行测试。

在[简单案例](#简单案例)中，我们准备了 `1000` 个问题打算进一步测试，这个就是数据集。在实际业务场景中，常用的数据集有 `CEval`, `BoolQ` 和 `HumanEval`，每个数据集针对不同的场景：

- `CEval`：单选题。不同学科为一个文件夹，问题和答案按照 `json` 格式储存在文件夹下。想办法让模型只回答 `ABCD`，然后计算模型的准确率 
- `BoolQ`：判断题。
- `HumanEval`：代码题

使用数据集测试来识别 `bad case` 的流程较为繁琐。

首先，让测试模型和标杆模型同时跑一遍数据集，收集他们的平均总分。如果测试模型的分数远远低于标杆模型分数，那么我们认为测试模型 **掉点严重**，存在精度问题。如果分数相差无几，那么我们认为模型不存在精度问题，不需要进一步定位。那么何为相差无几？不同的模型，在不同的业务场景下有自己的验收标准，根据验收标准来决定多大的分数误差算作精度问题。

如果模型掉点严重，那么我们需要对数据集中的问题进行**划分**，从而识别 `bad case`。为什么需要划分呢？因为，在数据集测试场景下，对于其中任意一个问题，除了存在标杆模型的答案和测试模型的答案以外，还有一个标准答案（ground truth or label）。我们通过标准答案来评定一个模型的输出是对还是错，然后计分。比如，在 `CEval` 单选题下，我们会让模型只返回 `A，B，C，D` ，从而和标准答案进行比较。虽然，我们单个模型的平均分是根据数据集的标准答案来进行衡量的，然而，精度对比，**是测试模型跟标杆模型进行比较，而不是测试模型跟标准答案进行比较**。

为了更好的阐述如何划分数据集，我们定义 `y_ref` 为标杆模型的输出，`y_test` 为测试模型的输出，`y_true` 为数据集标准答案。按照划分的优先级顺序，从高到低，一共会有四个场景，如下:

1. `y_ref == y_true and y_test != y_true`
2. `y_ref == y_true and y_test == y_true`
3. `y_ref != y_true and y_test != y_true`
4. `y_ref != y_true and y_test == y_true`

这里需要放一个图

#### 他对我不对
第一种情况是最贴切 `bad case` 的，也是我们的首选划分。这种情况下，对于问题 $x$，标杆模型的回答和标准答案一致，然而测试模型和标准答案不一致。由于我们希望测试模型的表现尽量靠近标杆模型，所以标杆模型答对的题目，我们要求测试模型必须答对。

因此，凡是标杆模型答对，但是测试模型没有答对的题目，$x$，都属于 `bad case`。划分的这类数据集叫做首选数据集。 

#### 咱俩都对了
第二种情况，并不属于 `bad case` 范畴，但是是精度比对中重要的一环。这种情况下，对于问题 $x$，标杆模型和测试模型的答案都和标准答案一样，那岂不是意味着测试模型的表现已经和标杆一样了吗？那划分出这类数据集的意义何在呢？

事实上，这类数据集可以被称为参考数据集。为了处理精度比对中的极端情况。回想一下，我们在 [简单案例](#简单案例) 中，是如何发现问题算子的？我们首先储存了所有中间结果，然后通过层层比对，发现了问题。然而这里，实际上默认使用了一个 *绝对误差* 的评测指标，即，我们将两个结果做了一个差，然后取绝对值。由于这个误差足够大（$|17 - 10| = 7$），那么根据常理或者经验，我们认为这么大的误差就代表精度问题。

在大模型场景下，我们的中间结果往往不是整数，而是小数，是浮点数。如果两个算子的 `out tensor`，在余弦相似度评测指标下，结果为 $0.857792$，我们应该认为它存在精度问题还是不存在精度问题呢？这是没有一个固定的评判指标的。对于经验老道的模型调优开发者而言，可能他们内心存在一杆秤，靠经验和直觉能够进行正确的评判。但是有没有一个可以信赖的标准呢？这个时候，我们的参考数据集就发挥了作用。

因为在参考数据集下，两个模型的结果均保持一致，且逻辑上都是对的，那么我们如果保存他们的中间数据，然后进行比较的话，就可以得到每个算子在每个不同的误差衡量指标下的一个 **正常** 范围。如果我们发现，在备参考数据集下，一个算子的余弦相似度的比较结果，最低可以到 `0.84`，那么当我们处理首选数据集中所有的 `bad case` 的时候，我们就对于所有低于 `0.84` 的情况都认为是精度不达标。

因此，参考数据集不是 `bad case`，但是有助于进行极端情况下的精度评测。

#### 其他
剩下两种场景都没有什么实际的参考价值。

## 比较和落盘
到这为止，我们已经知道了如果定位 `bad case`。下一步就是使用 `ait` 工具进行数据落盘和比较。

如何使用 `ait` 工具进行 [落盘]() 和 [比较]()，点击进行详细了解，这里只讲精度比对思路。

我们保存的所有中间数据，其实都是为了最后的比较而服务的。数据落盘的参数众多，考虑的场景也很全面，但是往往我们比较只用到了其中微不足道的一些资源。如果不清楚哪些是比较所需要的，而额外保存了很多其他的内容，对于整个精度比对流程而言是很没有效率的。

### 同一个 `token` 比较

如果我们想要比较测试模型的第 `3` 个 `token`，与标杆模型第 `3` 个 `token` 是否存在精度问题，实际上比较的，是两个模型在计算第 `3` 个 `token` 的所有中间数据。此时对于 `ait` 工具而言，是比较同一个 `token` 下面的每一个 `layer`。那么，`ait` 工具需要的资源有

1. 两个模型的拓扑信息，即 `model.json`
2. 两个模型在计算第 `3` 个 `token` 时的，每一层的 `tensor` 信息。目前工具只比较 `after tensor` 中的 `out tensor`。

那么根据 `ait llm compare` 的需求，我们可以使用 `ait llm dump` 的思路就很清晰了。如果两个模型都是 `atb` 模型，那么落盘命令如下：

```sh
`ait llm dump --exec <推理命令> --type model tensor -stp 1 -er 3,3 -o <保存地址> -l error
```

其中：
- `--type model tensor` 表示保存模型拓扑信息，每一层的拓扑信息和 `tensor` 信息。
- `-stp 1` 表示只保存 `out tensor`，不保存 `in tensor`
- `-er 3,3` 表示只保存第 `3` 个 `token` 的中间计算信息
- `--log-level error` 表示只有 `error` 或者更严重的日志才会打屏

`ait llm dump` 默认保存 `after tensor`。

如果有 `torch` 模型，那么不能使用命令行的方式进行数据落盘，得对源码进行插桩，如下：

```py
# ...
# model initialization
model = AutoModel.from_pretrained(...)

from ait_llm import DumpConfig, register_hook
dump_config = DumpConfig(token_range=[3], tensor_part=1, dump_path=<保存地址>)
register_hook(model, dump_config)

# model inference
# ...
```
其中：
- `token_range=[3]` 表示只保存第 `3` 个 `token` 的中间计算信息
- `tensor_part=1` 表示只保存 `out tensor`，不保存 `in tensor`

默认保存模型拓扑信息和 `tensor` 信息。

`DumpConfig` 是一些落盘相关的参数设置，`register_hook` 则通过 `torch` 自带的 `register_module_forward_pre_hook` 来进行插桩，[详情请戳](https://pytorch.org/docs/stable/generated/torch.nn.modules.module.register_module_forward_pre_hook.html)。

数据落盘之后，使用 `ait llm compare` 即可进行比对：

```sh
ait llm compare -gp <标杆模型第三个 token 目录> -mp <测试模型第三个 token 目录> -o <保存地址> -l error
```

| 3        | 1       | xxx              | ... |
| 3        | 2       | xxx              | ... |

由于 torch 模型拓扑和 atb 拓扑存在名称不一致的情况，ait 工具内部实现了一些常见的算子映射关系，如有特殊情况没有覆盖到的，通过 -mf 的方式加入，见 [1](https://gitee.com/ascend/ait/blob/master/ait/docs/llm/%E5%B7%A5%E5%85%B7-%E8%87%AA%E5%8A%A8%E6%AF%94%E5%AF%B9%E5%8A%9F%E8%83%BD%E4%BD%BF%E7%94%A8%E8%AF%B4%E6%98%8E.md)

### 多个 `token` 比较

除了比较同一个 `token` 每一层的计算结果之外，`ait llm compare` 还支持多个 `token` 同时比较。目前多个 `token` 自动比对仅适用于 `torch` 和 `atb` 比对场景。

具体的细节可以参考 [这篇文档](https://gitee.com/ascend/ait/blob/master/ait/docs/llm/%E5%8A%A0%E9%80%9F%E5%BA%93%E5%9C%BA%E6%99%AF-%E8%BE%93%E5%87%BAToken%E7%9A%84logits%E7%B2%BE%E5%BA%A6%E6%AF%94%E5%AF%B9.md#/ascend/ait/blob/master/ait/docs/llm/%E7%B2%BE%E5%BA%A6%E6%AF%94%E5%AF%B9%E7%BB%93%E6%9E%9C%E5%8F%82%E6%95%B0%E8%AF%B4%E6%98%8E.md)

token 变，data 不变

多个 `token` 比较并不常用，一般直接比较第 `0` 个 `token` 的内容，如果解决不了问了，再考虑多 `token`

注意：选择了 dump 第几个 token 不代表一定可以 dump 到那么多个 token，要根据 `max_new_tokens` 的设定来和模型推理的实际情况而定。

该案例以 `chatglm2-6b` 模型来详细介绍一下 `torch` 模型和 `atb` 模型的精度比对流程。该案例假设 `Where is the capital of China?` 是 `bad case`。以下称标杆模型为 `torch` 模型，称 `atb` 模型为测试模型。

## 准备
1. 模型权重文件
2. `torch` 模型相关配置文件
3. `atb` 模型相关配置文件

## 推理
在使用 `ait` 工具之前，首先必须检验 `torch` 模型和 `atb` 模型推理脚本是否能够正常运行。有时， `atb` 模型存在算子未正常接入的问题，一定要查看日志来进行排查。

`torch` 模型的推理推荐使用 `cpu` 进行推理。如下是一个模拟脚本 `run_torch.py`

```py
# run_torch.py
tokenizer = AutoTokenizer.from_pretrained("tokenizer_config", trust_remote_code=True)
model = AutoModel.from_pretrained("model_weights", trust_remote_code=True, torch_dtype=torch.float).eval()

with torch.no_grad():
    inputs = tokenizer(
                "Where is the capital of China?", # bad case input
                padding="max_length",
                return_tensors="pt",
                truncation=True,
                max_length=1024)

    outputs = model.generate(**inputs, do_sample=False, max_new_tokens=1)

print(tokenizer.decode(outputs.tolist()[0]))
```

读取模型，然后对于我们的 `bad case` 进行推理。最后输出推理结果。推理命令如下

```sh
$ python3 run_torch.py
Where is the capital of China? 北京
```

`atb` 模型的推理这里不做演示，详细请参考 `MindIE`。

## 落盘
确认推理过程无误之后，我们便可以使用 `ait` 工具来保存中间数据，进而进行比对，发现问题。

`torch` 模型的落盘方法需要修改源代码。在模型定位完之后，在执行推理之前，使用 `ait` 工具进行设置插桩。如下：

```py
# run_torch.py
tokenizer = AutoTokenizer.from_pretrained("tokenizer_config", trust_remote_code=True)
model = AutoModel.from_pretrained("model_weights", trust_remote_code=True, torch_dtype=torch.float).eval()

from ait_llm import DumpConfig, register_hook
dump_config = DumpConfig(dump_path="/home/torch_dump")
register_hook(model, dump_config)

with torch.no_grad():
    inputs = tokenizer(
                "Where is the capital of China?", # bad case input
                padding="max_length",
                return_tensors="pt",
                truncation=True,
                max_length=1024)

    outputs = model.generate(**inputs, do_sample=False, max_new_tokens=1)

print(tokenizer.decode(outputs.tolist()[0]))
```

上述脚本中，我们定义了落盘位置为 `/home/torch_dump`，在执行完毕之后，我们能够在 `/home/torch_dump` 文件夹下，找到我们保存的中间结果。

## Before and After Tensor
实际上，这里的 `after` 是关于推理过程而言的。举个例子，我们可以把每个算子理解为一个函数，考虑如下函数：
```py
def add_one_operation(ls):
    for i in range(len(ls)):
        ls[i] += 1
        
    return ls
```
这个函数很简单，就是将传入的列表中所有的元素加上 `1`。

现在我们来模拟 `dump` 采集的场景。假设在执行 `add_one_operation([1, 2, 3])` 这个命令之前，我们采集一下数据，我们可以得到：

<center>

| param / inTensor | retval / outTensor |
| :---: | :---: |
| [1, 2, 3] | undefined |

</center>

此时函数输入已经拿到，为 `[1, 2, 3]`，输出还没有得到，因为我们还没有执行这个函数。

如果在函数执行完毕之后采集，我们可以收集到

<center>

| param / inTensor | retval / outTensor |
| :---: | :---: |
| [2, 3, 4] | [2, 3, 4] |

</center>

发现了什么？此时输入和输出结果一致了，因为函数本身是对传入数据进行了原地修改（in-place）。那么同理，如果一个算子是原地修改一块内存的数据，那么在推理完之后再进行信息采集，就会得到输入和输出相同的情况。这在某些场景下，可能会导致精度定位的困难。然而这种情况特别不常见。


模型在量化后，如果单精度出现问题，首先想想是不是环境的问题，或者是 modelslim 参数是不是需要调整。如果实在定位不到问题，ait llm dump will be our last resort。但是在 torch 到加速库模型中，如果出现精度问题，感觉还是先问下模型迁移的人，是不是存在映射错误，如果定位不到问题，然后再 ait llm dump
