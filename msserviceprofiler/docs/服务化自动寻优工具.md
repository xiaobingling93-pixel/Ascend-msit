# 服务化自动寻优工具
## 基本功能
- llama 等模型部署时，用户经常会对服务化的性能有要求，人工寻优耗时耗力，并且长时间占用用户机器。服务化参数众多，不同参数组合对性能影响较大，且无显然的线性规律。
- 服务化自动寻优工具支持对mindie和vllm进行自动寻优，获取符合要求的最佳参数组合。自动寻优过程不依赖于npu推理

| 版本配套关系 |     CANN     |     框架     |
|:-------------:|:------------:|:--------------:|
|     mindie当前版本      | CANN 8.1.RC1 | MindIE 2.0.RC1 |
|     vllm当前版本      | CANN 8.2.RC1 | vllm 0.8.4 |
## 工具原理
工具使用xgboost来模拟真实的模型推理，通过xgboost预测获得推理时间，从而节省这部分时间以及npu的占用，但是需要用户在寻优前按下面步骤训练xgboost模型。
同时由于前置步骤较多，工具也提供一种更为简单的轻量化使用方式，不需要训练模型，直接在真实推理的基础上通过算法对参数进行寻优。目前工具已基于llama3-8b通过验证，
理论上不限制支持模型范围，后续计划扩大支持范围的验证。
# 使用方式
## 安装依赖
```
cd msit/msserviceprofiler
pip install -U msserviceprofiler .[speed] # 安装相关加速包，提高训练效率。
```
```
如果上述安装失败，可尝试安装较少依赖的三方包，但训练模型时，大数据量时性能较低。
pip install -U msserviceprofiler .[train] # 安装训练依赖。
```
```
如果使用轻量化的方式进行寻优则只需安装最少的依赖即可
pip install -U msserviceprofiler .[real] # 安装基本依赖。
```
## 训练模型
参照[服务化调优工具手册](https://www.hiascend.com/document/detail/zh/mindstudio/80RC1/T&ITools/Profiling/mindieprofiling_0001.html) 开启profiling实际跑一遍mindie推理服务的测试脚本，将采集的profiling数据进行解析然后用于训练模型。profling采集数据需要包括batch_type, batch_size, forward_time, batch_end_time(ms), request_recv_token_size, request_reply_token_size, need_blocks, request_execution_time(ms), first_token_lantency(ms)

```
cd msit/msserviceprofiler/msserviceprofiler/modelevalstate
msserviceprofiler train -i=/path/to/input -o=/path/to/output 
```
|参数|说明|是否必选|
|---|---|---|
|-i, --input|输入数据目录，这里所需的数据即profiling的输出路径。|是|
|-o, --output|输出目录，建议输出在modelevalstate下面创建一个/result/latency_model目录来存放。若未指定则会在当前目录下生成。|否|
|-t, --type|框架类型，可选mindie/vllm，默认mindie。|否|
## 修改配置文件
启动寻优前需用户按照实际情况配置run_benchmark.sh，可以参照[mindie benchmark 输入参数说明](https://www.hiascend.com/document/detail/zh/mindie/20RC1/mindieservice/servicedev/mindie_service0153.html)。(vllm使用run_vllm_benchmark.sh)
### config.json参数说明
用户可以在config.json中进行设置参数来按照实际需求对寻优的性能指标进行限制。
|参数|说明|
|---|---|
|n_particles|寻优种子数，即一组生成的参数组合数。|
|iters|迭代轮次数。|
|prefill_lam|time_to_first_token即首token时延超时惩罚系数，若对time_to_first_token没有时延要求设置为0即可。|
|decode_lam|time_per_output_token即非首token时延超时惩罚系数，若对time_per_output_token没有时延要求设置为0即可。|
|prefill_constraint|time_to_first_token的限制时延。|
|decode_constraint|time_per_output_token的限制时延。|
|service|标注多机启动时为主机或从机|
|target_field|对应mindie的config.json中的参数，详情可以参考[mindie server 配置参数说明](https://www.hiascend.com/document/detail/zh/mindie/20RC1/mindieservice/servicedev/mindie_service0285.html)。(vllm框架则为vllm_config.json中的参数)|
## 启动寻优
- 寻优时需开启环境变量export MODEL_EVAL_STATE_ALL=True。
- 若采用不依赖profiling的方式进行寻优需开启环境变量export MODEL_EVAL_STATE_IS_SLEEP_FLAG=true。
```
msserviceprofiler optimizer -lb -d multiple --backup
```
|参数|说明|是否必选|
|---|---|---|
|-lb, --load_breakpoint|控制是否从断点恢复寻优过程，默认关闭。|否|
|-d, --deploy_policy|设置部署策略，即单机或多机部署，默认为单机部署。若为多机参数需修改为multiple|否|
|--backup|决定是否在寻优过程中备份数据，默认False，不开启，用法例如--backup。|否|
|-b, --benchmark_policy|可选：benchmark/profiler_benchmark/vllm_benchmark，默认为使用mindie的benchmark，vllm框架下需要使用vllm_benchmark|否|
### 轻量化寻优
轻量化寻优只需安装最少的依赖并且不需要训练模型，直接开启寻优即可：
```
cd msit/msserviceprofiler/msserviceprofiler/modelevalstate
msserviceprofiler optimizer -lb -d multiple --backup
```
以上参数均为可选，若不需要开启这些功能则直接使用
```
msserviceprofiler optimizer
```
## 多机部署
多机部署（目前只支持双机部署）时，需要用户在两台机器上创建一个共享目录并在两台机器中的config.json中添加communication字段如：
```
{
    "communication": {"cmd_file": "/common/cmd.txt", "res_file": "/common/res.txt" }
}
```
修改从节点上的config.json的service值为slave，然后启动服务
```
msserviceprofiler optimizer
```
vllm框架下暂不支持多机部署。

# 输出结果
## 输出文件说明
可以使用export MODEL_EVAL_STATE_CUSTOM_OUTPUT来设置输出路径。
输出件为csv，若不指定输出路径则在modelevalstate/result/store下存放。
输出csv中的每一行对应一组参数，前四列为性能指标。用户可以根据需求筛选满足要求的性能行，将mindie参数以及benchmark的参数改为csv中的数据即可。
- generate_speed列为吞吐。
- time_to_first_token列为ttft时延，单位为秒。
- time_per_output_token列为tpot时延，单位为秒。
- success_rate列为测试返回请求成功率。
- CONCURRENCY列为benchmark的并发数。
- REQUESTRATE列为benchmark的发送速率。
- error列为记录这次参数没有正常执行的原因，在发送错误时记录。
- backup列为数据记录地址，当开启--backup时记录。
- 其余列为对应的mindie的config.json参数（vllm其余列为对应的vllm_config.json）。
