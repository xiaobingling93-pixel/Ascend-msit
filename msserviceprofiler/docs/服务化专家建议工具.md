# MindStudio Service Profiling Advisor 专家建议工具

## 概述

大模型推理服务化框架的各类性能问题场景较为复杂，对于没有丰富调优经验的用户而言，性能指标分析与推理调优较为困难。本工具基于MindIE-Service服务化框架，及benchmark的输出结果，可以一键式给出调参建议，提供给用户作为调优参考。

## 介绍
### 基本功能

本工具基于 ① benchmark 的 instance 输出结果，② MindIE-Service 的 config.json 配置，③ npu显存情况，④部署模型等信息，综合分析给出 config.json 中 maxBatchSize、maxPrefillBatchSize 等配置的调参建议，用于提高 TFTT / Throughput 等性能优化指标。

**注意：由于 CPU、内存等硬件差异，网络环境等不同，以及模型参数配置的细节不同等，当前建议值不能保证性能一定有提升，需要实际修改后验证**


### 工具安装
**pip 安装**
  ```sh
  pip install -U msserviceprofiler
  ```
  
**离线安装**
  - 在能够访问网络的机器上，访问 [PyPI 官方源](https://pypi.org/project/msserviceprofiler/)，点击左侧 `Download files`下载
  - 下载完成后，上传到服务器中
  - 假设 wheel 包存放路径为 `whl_path`，输入下列命令进行安装
      ```bash
      pip install whl_path
      ```
  - 终端输入 `msServiceProfiler` 校验是否安装成功


### **使用方式**
- **数据依赖**

  ① mindie benchmark 结果输出正常，生成的instance文件夹放置路径为${path}/instance/

  ② mindie server config.json文件配置正确，一般位于/usr/local/Ascend/mindie/latest/mindie-service/conf目录下

  ③ 在服务化进程拉起前（即mindie server config.json中配置的npuDeviceIds卡号，其剩余可用显存不能为0）执行下述命令行；或者显式得指定mindie server config.json中的npuMemSize

- **执行命令行**
  ```sh
  msServiceProfiler advisor -i ${path}/instance/
  ```
- **如果希望通过源码方式使用（需在工具安装后）**
  ```sh
  git clone https://gitee.com/ascend/msit.git
  cd msit/msserviceprofiler
  export PYTHONPATH=$PWD:$PYTHONPATH
  python msserviceprofiler/__main__.py advisor -i ${path}/instance/
  ```

- **命令行参数**

  | 参数                 | 说明                                                            |是否必选|
  | -------------------- | --------------------------------------------------------------- |-----|
  | -i, --instance_path  | benchamrk 输出的 instance 路径，不输入则默认不读取其中相关信息用于分析  |否|
  | -s, --service_config_path  | MINDIE service 路径或 config json 文件路径，默认值为环境变量中的 `MIES_INSTALL_PATH`，如果均未配置则使用 /usr/local/Ascend/mindie/latest/mindie-service |否|
  | -t, --target         | 调参指标, 可选值：ttft, firsttokentime, throughput，默认ttft       |否|
  | -m, --target_metrics | 调参指标的具体项，可选值：average,max,min,P75,P90,SLO_P90,P99,N，默认average |否|
  | -l, --log_level  | 日志级别，可选值 debug,info,warning,error,fatal,critical，默认info  |否|
  | -in, --input_token_num  | 可选参数，请求输入长度，需为正整数。不输入则默认从benchmark的instance结果中获取 |否|
  | -out, --output_token_num  | 可选参数，请求输出长度，需为正整数。不输入则默认为mindie service config.json的maxIterTimes值 |否|
  | -tp, --tp  | 可选参数，tp域大小，需为正整数。不输入则默认从mindie service config.json文件中获取，未取到则默认为1 |否|


### **结果说明**
- **建议结果1：**
  - 输入 -i 参数或 -in 参数，mindie server config.json中相关参数配置正确，并且满足服务化配置的可用npu显存不为0（即上述“使用方式”中“数据依赖”第③点），有下述建议结果。
  - 表示根据当前可用显存信息、模型结构信息、mindie service config.json配置信息及请求输入输出长度等情况综合考虑，计算出当前mindie service config.json中的maxBatchSize的建议取值范围。
  - 建议用户根据下述提示，将mindie service config.json中的maxBatchSize设置为平均值大小，maxPrefillBatchSize设置为maxBatchSize的一半，重新拉起服务化运行，观察性能是否提升。
  - 若性能有提升，则可尝试逐渐向range中的最大值靠近，观察性能指标变化情况。
  - 若maxBatchSize值设置过大，则会导致模型拉起失败，此时应当将该值向range中的最小值靠近，直到模型能成功拉起。
  - maxPrefillBatchSize通常都设置为maxBatchSize的一半。

  ```sh
  # msservice_advisor_logger - INFO - </think>
  # msservice_advisor_logger - INFO -
  # msservice_advisor_logger - INFO - <advice>
  # msservice_advisor_logger - INFO - [config] maxBatchSize
  # msservice_advisor_logger - INFO - [action] 取值范围为 [xx, xx]，平均值为 xx
  # msservice_advisor_logger - INFO - [reason] 经过对当前显存信息的计算，建议将maxBatchSize的值设置为平均值大小，并逐渐向范围最大值调整，以占满整个显存
  # msservice_advisor_logger - INFO - </advice>
  ```
- **建议结果2：**
  - 根据 `-i, --instance_path` 指定的 benchmark 输出 `instance` 结果，拟合数据并给出建议值，该结果中 `maxBatchSize` 会综合考虑根据 NPU 现存大小计算的结果
  - **可尝试将该推荐值应用到 `config.json` 中，并重新验证性能查看是否有所提升，如果结果不理想，可尝试只应用其中一个值，并验证推理性能**
