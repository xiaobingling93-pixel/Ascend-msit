# 负载均衡亲和专家寻优

负载均衡工具 (Expert Load Balancing Tools) 是在Deepseek模型静态/动态场景下进行负载均衡亲和专家寻优。通过加载模型推理时 dump 到的专家热度信息文件（格式为csv），经过算法计算后落盘 prefill 或 decoder 层的配置文件。

#### 工具安装
需要安装msit工具，软件安装见 [负载均衡亲和专家寻优工具安装](../install/README.md)。
安装好msit后需要安装 msit 中的 elb 组件，执行 msit install elb。

#### 版本要求

其中算法 speculative moe 依赖 ortools==9.6.2534。
负载均衡亲和专家寻优工具使用python 3.9及以上版本

####   算法介绍

目前提供两种场景下的算法：
- 静态场景 + Atlas 800I A2 推理服务器：计算通信负载均衡算法（C2LB）和 speculative moe interface 算法
- 静态场景 + Atlas 800I A3 推理服务器：speculative moe interface 算法
- 动态场景 + Atlas 800I A2 推理服务器：计算通信负载均衡算法（C2LB）生产初始配置文件

###  算法介绍

#### C2LB算法

###### 要求冗余专家数量需要小于等于NPU的数量

对于MoE类模型，在使用专家并行（EP）时，不同的专家会被分配到不同的 GPU/NPU 上，由于不同专家的负载可能会根据当前工作负载而变化，因此保持不同 GPU/NPU 的负载均衡非常重要。
C2LB（Compute Communication Load Balance）计算通信负载均衡算法是一种静态专家放置策略，基于离线统计的专家负载信息，全局考虑计算通信均衡，输出专家在卡上的放置策略。该算法支持冗余专家放置，支持卡上专家数量不同的部署，当前约束卡的数量大于等于冗余专家数量，且每个卡只能部署一个冗余专家。受限于数据集离线统计的专家负载信息的偏差，静态专家部署方案难以应对负载变化大的场景。


#### speculative moe interface 算法

speculative-moe 算法适用于推理系统正式运行之前、初始化专家部署阶段，通过调用该算法，可基于专家热度信息，生成全局专家部署json文件；算法通过原始+冗余专家布放聚合求解，实现计算/通信拖尾消减，提升系统有效吞吐；优势场景为64卡、全局320专家（256原始专家+64冗余专家），每设备5专家，单卡batrch size 大于等于16的场景。

- 静态场景下目前适用于 A2 的算法默认的计算求解时间为300s，后续会开放用户自主设置求解时间。
- 静态场景下目前适用于 A3 的算法默认的计算求解时间为900s，后续会开放用户自主设置求解时间。
