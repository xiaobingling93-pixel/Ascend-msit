# 负载均衡亲和专家寻优

msit elb 工具是在Deepseek模型静态场景下进行负载均衡亲和专家寻优。通过加载模型推理时 dump 到的专家热度信息文件（格式为csv），经过算法计算后落盘 prefill 或 decoder 层的配置文件。

#### 工具安装
需要安装msit工具，软件安装见 [负载均衡亲和专家寻优工具安装](../install/README.md)。
安装好msit后需要安装msit中的expert-load-balancing组件，执行 msit install expert-load-balancing。

#### 版本要求

其中算法 speculative moe 依赖 ortools==9.12.4544
负载均衡亲和专家寻优工具使用python 3.9及以上版本

####  使用说明

目前提供两种场景下的算法：
- 静态场景 + Atlas 800I A2 推理服务器：计算通信负载均衡算法（C2LB）和 speculative-moe-level 1 和 speculative-moe-level 3 算法
- 静态场景 + Atlas 800I A3 推理服务器：speculative-moe-level 1 和 speculative-moe-level 3 算法
- 动态场景 + Atlas 800I A2 推理服务器：计算通信负载均衡算法（C2LB）生产初始配置文件

本工具所需的负载均衡专家热度信息文件以来mindie提供的dump能力，需要使能两个mindie的环境变量来获得输入文件，保证在纯模型或服务化场景，采用实际务数据或数据集，能正常完成模型推理之后设置export MINDIE_ENABLE_EXPERT_HOTPOT_GATHER=1，设置export MINDIE_EXPERT_HOTPOT_DUMP_PATH=“用户自定义路径”。

如果使用mindie RC1 B080及之前版本进行输入文件采集，dump 下的 decode 和 prefill 文件会保存在一个文件夹中，使用speculative moe level_1 和 speculative moe level_2 算法时需要手动将 prefill 和 decode 的文件分别存放到两个文件夹中，分两次调用工具生成 decode 和 prefill 的配置文件。而mindie RC1 B080之后的版本采集到的 prefill 和 decode 文件会自动保存在不同的文件夹下，无需手动拆分。

###  算法介绍

#### C2LB算法

###### 要求冗余专家数量需要小于等于NPU的数量

对于MoE类模型，在使用专家并行（EP）时，不同的专家会被分配到不同的 GPU/NPU 上，由于不同专家的负载可能会根据当前工作负载而变化，因此保持不同 GPU/NPU 的负载均衡非常重要。
C2LB（Compute Communication Load Balance）计算通信负载均衡算法是一种静态专家放置策略，基于离线统计的专家负载信息，全局考虑计算通信均衡，输出专家在卡上的放置策略。该算法支持冗余专家放置，支持卡上专家数量不同的部署，当前约束卡的数量大于等于冗余专家数量，且每个卡只能部署一个冗余专家。受限于数据集离线统计的专家负载信息的偏差，静态专家部署方案难以应对负载变化大的场景。


#### speculative moe 算法

speculative-moe-level 1、speculative-moe-level 2 算法适用于推理系统正式运行之前、初始化专家部署阶段，通过调用上述算法，可生成全局专家部署表；通过使能该专家部署表，可实现EP通信/专家计算负载均衡，提升系统有效吞吐。
level 1/level 2 算法输入均为离线采集的多阶段专家激活信息，其中，level 1 算法可在亚分钟时间内完成运行，获得次优解；level 2 算法采用算法增强，level 1/level 2算法均支持A2/A3推理系统Decoding/Prefill阶段。其中：

1）	A2-D阶段，优势场景为8机64卡EP，部署320专家（256原始+64冗余专家），每卡5专家，单卡bs不小于16；

2）	A3-D阶段，优势场景为：a）18机144卡EP，其中2机16卡部署共享专家，其余16机128卡部署512非共享专家（256原始+256冗余专家），每卡4专家，单卡bs不小于64；b）4机32卡EP，部署320专家（256原始+64冗余专家），每卡10专家，单卡bs不小于32；

3）	A2-P阶段，单实例卡数为16/32，优势场景为每个实例下总冗余专家数量不超过1*单实例卡数；

4）	A3-P阶段，单实例卡数为8/16，优势场景为每个实例下总冗余专家数量不超过2*单实例卡数。
